梯度表示图像亮度在 x/y 方向上变化有多快，代码里就是先做 Sobel(dx, dy)，再算幅值和角度，line2Dup.cpp。
所谓“轮廓”其实就是梯度幅值很大的地方。所以模板特征大多落在物体边界附近，但本质来源仍然是原图梯度，不是独立的 contour 数据。
如果是彩色图，这份代码会在三个通道里选“梯度最强的那个通道”的方向，line2Dup.cpp。


一个很容易混的点是：

轮廓方向是沿着边界走的方向。
梯度方向是垂直于边界的方向，指向“亮度增加最快”的方向。


比如一条竖直边界，真正用来匹配的不是“边界是竖着的”，而是“这条边左右亮度变化的方向”。
所以这篇方法用的是边界法线方向，不是边界切线方向。


响应度 / response”可以理解成：
在待测图里，某个位置对某个方向“像不像”的分数。
代码先把待测图方向量化，再做 spread，把一个小邻域里出现过的方向扩散开，line2Dup.cpp。
然后为 8 个方向分别生成响应图，line2Dup.cpp。
模板里如果某个点要求方向是 ori，匹配时就去 ori 那张响应图上读值，再把所有模板点的值加起来，line2Dup.cpp。
最终归一化成相似度分数，line2Dup.cpp。

是这 8 个“无符号梯度方向”：

0: 0° / 180°
1: 22.5° / 202.5°
2: 45° / 225°
3: 67.5° / 247.5°
4: 90° / 270°
5: 112.5° / 292.5°
6: 135° / 315°
7: 157.5° / 337.5°
代码里先把 360° 量化成 16 桶，再把相差 180° 的方向折叠成 8 桶，所以不区分“黑到白”和“白到黑”，只区分“边的朝向”。对应实现见 line2Dup.cpp:357。
意思是：把“有方向的梯度”变成“无方向的朝向”。

先看一个最直接的例子。

情况 A：黑 -> 白
0   0   0   255 255
梯度指向右边，角度约 0°

情况 B：白 -> 黑
255 255 255 0   0
梯度指向左边，角度约 180°
这两个梯度方向 0° 和 180° 在数学上不同，但它们描述的其实是同一类边：
都是一条竖直边界，只是亮暗两侧反过来了。

所以论文/代码不想区分：

黑到白
白到黑
只想区分：

这是竖直边
还是水平边
还是 45° 斜边
这就是“只区分边的朝向”。
代码里怎么做的

在 line2Dup.cpp:361：

angle.convertTo(quantized_unfiltered, CV_8U, 16.0 / 360.0);
含义是把 0°~360° 切成 16 桶，每桶 22.5°。

大概是这样：

0 号桶: 0°~22.5°
1 号桶: 22.5°~45°
...
8 号桶: 180°~202.5°
...
15 号桶: 337.5°~360°
然后在 line2Dup.cpp:380：

quant_r[c] &= 7;
这相当于只保留低 3 位，也就是：

0 和 8 合并
1 和 9 合并
2 和 10 合并
...
7 和 15 合并
也就是：

0° 和 180° 归为同一类
22.5° 和 202.5° 归为同一类
45° 和 225° 归为同一类
...
于是 16 类变成 8 类。

为什么这么做

因为对形状匹配来说，更重要的是边在哪里、朝哪个方向摆着，而不是：

左边黑右边白
还是左边白右边黑
这样做会更稳，对光照反转、对比度变化更不敏感。

一句话版

360° -> 16 桶：先粗量化梯度方向。
16 桶 -> 8 桶：再把相差 180° 的方向当成同一种，忽略亮暗极性。

# 提速
这篇文档里的提速其实是一整套组合拳，不只是你刚才说的 cache-friendly / 线性内存布局 这一招。

大致可以分成这几类：

1. 稀疏采样搜索

文档里一个很关键的点是：因为做了 orientation spreading，匹配对小平移有容忍度，所以没必要每个像素都算，只需要每隔 T 个像素评估一次。
这相当于把搜索位置从整张图，降成原来的大约 1 / T²。如果 T=8，理论上候选位置数量就是原来的 1/64。

这其实是非常大的提速来源，往往比“纯底层优化”更重要。

2. 用二值表示 J 取代每次现算局部最大值

原始公式里，对模板中每个特征点，都要在一个邻域里找“最相近的方向”，这一步如果每次都现场做 max 搜索，会很慢。

论文的做法是：

先把方向量化

再把一个局部邻域里出现过哪些方向，编码成一个 bit string

得到一个二值表示 J

这样后面就不用反复在邻域里搜索最大值了，而是把这个 bit string 当成索引去查表。

这个属于算法级预处理换速度。

3. 预计算 response maps

在有了 J 之后，论文进一步把“某个方向在每个位置上的最佳响应”预先算成 S_i 这些 response maps。
这样真正匹配模板时，就不再做复杂比较了，而是变成：

直接从 response map 里取值，然后累加。

也就是说，把原来的：

比角度

取最大值

再累加

变成：

读表

加和

这一招是核心提速点之一。

4. 模板本身做稀疏化

论文不是拿整张模板的所有梯度点去匹配，而是只保留：

少量最有判别力的梯度方向点

也就是只选“最 discriminative”的那些点。
这样模板里的特征点数量 G 变小，匹配时每个模板要加的项也更少。

这个本质上是：

减少模板计算量。

5. 方向量化

它把连续梯度方向先量化成少量方向 bin。
这样做有两个好处：

可以用 bit 编码表示局部方向集合

可以用查表方式快速算相似度

如果不量化，而是每次都用连续角度做 cos 比较，计算量会大很多。

所以量化不仅是为了稳健性，也是为了让后面的查表和预计算成为可能。

6. SSE 并行

这就是你前面问到的那部分。
把 response maps 线性化之后，就可以用 SSE 一次并行处理 16 个 8-bit 值，而不是一个一个加。

这个属于：

底层实现优化 / SIMD 并行优化

它很重要，但它是在前面那几层算法优化完成之后，再进一步压榨 CPU 性能。

7. 用 byte 近似响应值

为了让 SSE 更高效，论文还提到把 lookup table 里的响应值近似成 byte，这样才能更容易做到一次并行处理 16 个值。

这算是一个比较工程化的提速手段：

用较低精度换更高吞吐。

哪些是“最大头”的提速

如果按重要性来排，通常可以这么看：

第一层大头：

只每隔 T 个像素搜索

response maps 预计算

模板稀疏化

这些是算法级降复杂度，收益最大。

第二层加速：

cache-friendly 线性内存

SSE 并行

byte 近似

这些是实现级提速，是在已有算法框架上继续提速。

还有一个要区分

文档里还有一些内容不是为了提速，而是为了增强鲁棒性，比如：

spreading orientation

3D surface normal 模态

局部 3x3 众数量化抗噪

它们虽然有时也间接帮助减少模板数或搜索量，但主要目标是：

提高稳健性和判别性，不是单纯提速。

一句话总结：

有，而且不止一种。论文真正的速度优势来自“稀疏搜索 + 预计算 + 查表 + 模板稀疏化 + cache/SSE”的组合，而不是单靠缓存优化。
